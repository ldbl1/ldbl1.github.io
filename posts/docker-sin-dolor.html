Docker sin dolor: contenedores útiles para proyectos reales

Cómo uso Docker cuando quiero que un proyecto sea fácil de ejecutar, probar y mantener.

30 jun 2026#Docker#contenedores#DevOps#automatización

Por qué me gusta usar Docker

Docker me parece especialmente útil cuando un proyecto tiene dependencias, servicios auxiliares o versiones concretas que no quiero dejar “a la memoria” de quien lo instala. Para mí, un buen contenedor no es solo una forma de ejecutar una aplicación: es una manera de documentar cómo vive esa aplicación.

En proyectos de automatización, esto evita una de las frases más peligrosas: “en mi máquina funciona”. Si una automatización necesita una versión concreta de Python, librerías específicas, un navegador headless o una base de datos auxiliar, prefiero que todo quede expresado en ficheros versionados.

Mi enfoque mínimo

Intento empezar simple. Si el proyecto no lo necesita, no monto una arquitectura enorme. Normalmente separo tres piezas: el código, la imagen y la forma de ejecutarlo. El objetivo es que cualquier persona pueda clonar el repositorio y levantar el entorno con pocos comandos.

docker build -t mi-automatizacion .
docker run --rm mi-automatizacion

Cuando aparecen varios servicios, paso a docker compose. Por ejemplo, una automatización que necesita una base de datos local, un servicio mock o un panel de pruebas queda mucho más clara si todo está descrito en un único fichero de composición.

Qué suelo meter en el Dockerfile

Me gusta que el Dockerfile sea explícito, pequeño y fácil de leer. Una imagen que nadie entiende se convierte en otra caja negra. Como norma general, intento fijar versiones, copiar solo lo necesario y separar instalación de dependencias y copia del código.

FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .
CMD ["python", "main.py"]

Buenas prácticas que me han ahorrado problemas

La primera es no guardar secretos dentro de la imagen. Las credenciales deben entrar por variables de entorno, ficheros ignorados o mecanismos seguros del entorno donde se ejecute. La segunda es cuidar el tamaño de la imagen: menos dependencias suelen significar menos superficie de error.

También intento que los logs salgan por consola. En automatización, poder revisar qué ha pasado sin entrar manualmente al contenedor es clave para operar bien. Si una ejecución falla, quiero saber en qué paso, con qué entrada y con qué salida.

Conclusión

Para mí, Docker merece la pena cuando reduce incertidumbre. Si el contenedor hace que el proyecto sea más fácil de ejecutar, probar, desplegar o mantener, adelante. Si solo lo uso porque “queda moderno”, probablemente estoy resolviendo un problema que no existe.
